CS143笔记

至今为止已经或多或少学了一些编译原理相关的知识。我决定趁自己还记得它们的时候将它们记录下来。

Compiler Structure

编译一般分为5个阶段，分别是:

  • Lexical Analyze
  • Parsing
  • Syntax Analyze
  • Optimization
  • Code Generation

Lexical Analyze

总结一下就是这部分是用来对Token进行分类的。

比方说:

  • if是一个关键字，而i不是，它只是一个合法变量名。
  • 0x12是一个十六进制常量，而0是一个十进制常量。
  • a2ads8fa72是一个合法变量名，而2ads8fa72不是。

Finite Automaton

为了描述这种分类规则，我们可以使用名为Finite Automaton的数学工具。

A finite-state machine (FSM) or finite-state automaton (FSA, plural: automata), finite automaton, or simply a state machine, is a mathematical model of computation. It is an abstract machine that can be in exactly one of a finite number of states at any given time.

(来自Wikipedia)

其实之前在学习计算机网络的时候，描述TCP可靠传输的时候就已经使用过自动机来描述状态之间的转移。也算是一种预习吧（笑）

说句题外话，一直很佩服前人，连一个模式匹配都能和数学理论挂钩。对于数学苦手的我来说可能就和理论研究无缘了吧。

Regular Expression

不得不说把正则表达式和有限自动机放在一起教是非常天才的教法。相比网上的垃圾RE教程（像是直接告诉你什么代表什么含义的那种），显然这种教法更加容易让人学会并理解并应用正则表达式。

这就是为什么它是Nondeterministic的。 我猜想，对于比较复杂的模式，保持多个状态可能会造成效率严重低下（比方说需要保存的状态数以指数级增长的极端情况?），因此才使用DFA来进行模式匹配。不过这终究是猜想，转换的原因还需要进一步的了解。

DFA

全名是Deterministic Finite Automata，和NFA的区别只在于第一个单词。其实从字面上就可以大致了解它和NFA的不同: 它每次只能从一个状态转移到另外一个确定的状态。

神奇的是，我们可以把NFA转换为DFA。虽然通过详细的例子我已经明白转换步骤，但是回想起来我依然会惊叹于这种转换方法的奇妙。也许我依然还是只是流于表面，没有参透这种转换方法的核心思想。
